# 离线评测

这套评测用于回答三个问题：

1. 系统能否正确区分客服邮件和平台通知、验证码、营销邮件等非客服邮件？
2. 客服邮件的分类、风险等级是否符合人工标注？
3. RAG 返回的前三个知识来源中是否包含正确答案？

## 快速运行

只评测本地规则，不调用 LLM 或 Embedding API：

```powershell
cd backend
python evaluation/evaluate.py
```

评测当前数据库中的真实 RAG 检索：

```powershell
cd backend
python evaluation/evaluate.py --with-rag
```

`--with-rag` 会使用项目当前配置的 Embedding 模型，可能产生少量 API 费用。它不会生成回复，也不会发送邮件。

报告写入 `evaluation/reports/`：

- `baseline-rules.json`：完整机器可读结果和逐样本预测。
- `baseline-rules.md`：本地规则基线摘要。
- `baseline-rag.json`：包含 RAG 指标的完整结果。
- `baseline-rag.md`：RAG 基线摘要。

## 指标定义

- **客服识别 Precision**：系统判为客服邮件的样本中，真实客服邮件所占比例。
- **客服识别 Recall**：真实客服邮件中，成功进入客服链路的比例。
- **客服识别 F1**：Precision 与 Recall 的调和平均。
- **分类 Macro-F1**：对退款、账单、技术、产品咨询和投诉五类分别计算 F1 后取平均。
- **RAG Recall@K**：前 K 个去重知识来源中是否至少包含一个正确来源。
- **MRR**：第一个正确来源排名倒数的平均值，越接近 1 表示正确依据越靠前。
- **Precision@3**：前三个来源槽位中正确来源所占比例，用于观察无关上下文比例。

数据集是人工标注的脱敏场景，不读取真实邮箱，也不会把客户数据发送到评测之外。

## 当前基线

数据集当前包含 200 条样本，其中客服邮件 140 条、非客服邮件 60 条，
中英文各 100 条；五种客服分类各包含 28 条样本。

| 指标 | 结果 |
| --- | ---: |
| 客服识别 F1 | 94.89% |
| 分类 Macro-F1 | 67.82% |
| 风险 Macro-F1 | 94.23% |
| 风险 Macro-F1（校准集） | 93.63% |
| 风险 Macro-F1（留出集） | 95.96% |
| 语言识别 Accuracy | 100.00% |
| RAG Recall@1 | 95.71% |
| RAG Recall@3 | 99.29% |
| RAG MRR | 0.9714 |
| RAG 平均耗时 | 613.5 ms |
| RAG P95 耗时 | 453 ms |

当前主要改进方向：

1. 风险等级已从分类规则中解耦；下一步应补充真实脱敏误判样本，持续验证
   资金争议、账号安全、生产故障和投诉升级的边界。
2. 非客服过滤对调查问卷、服务条款通知等边界样本仍有少量误判。
3. 投诉类邮件涉及退款争议、取消服务和 SLA 时，首位结果偶尔被普通投诉政策占据。

重排优化前的 200 条对照报告保存在 `reports/before-rerank/`。优化前
RAG Recall@1 为 71.43%，优化后为 95.71%；对照报告必须保留，避免后续
只看到最终分数而无法判断改动收益。

风险判断最初的 Macro-F1 为 52.30%。引入独立风险评分器与 LLM 不可降级
护栏后提升到 80.92%；进一步修复英文子串误命中、区分产品能力咨询与真实故障，
并补充资金争议、生产故障和升级投诉的组合证据后，提升到 94.23%。评测按样本
ID 的 SHA-256 固定划分校准集与留出集，规则调整只参考校准集，留出集用于最终
泛化验证。

修改分类规则、提示词、知识库内容、Embedding 模型或重排权重后，应重新运行
两种模式，并将新报告与这份基线对比。
